1 · What Phase 9 is, and why it matters

Phases 0–8 took us from an idea to a live app. But shipping is the start of a product's life, not the end. Phase 9 is the never-ending loop that follows: gather feedback, fix bugs, add features, watch the system's health, pay down shortcuts, and repeat — with discipline instead of panic.

The three deliverables of this phase, in plain terms:
  • A backlog — an honest, prioritised list of everything we could do next, so work is chosen deliberately rather than by whoever shouts loudest.
  • A changelog + versioning — a human-readable record of what changed in each release, and a numbering scheme so "which version is running?" always has a clear answer.
  • Monitoring — a way to know the live app is healthy before a user emails to say it's broken.

Why can't we just "be done"?

Because the world around the app keeps moving: users find bugs and want features, dependencies get security patches, servers restart, data grows. Unmaintained software rots — it doesn't stay still. Maintenance is the cost of having something real and used.

2 · The iteration loop

Agile teams work in short, repeating cycles rather than one giant plan. Each loop ships a small improvement, learns from it, and feeds the next loop.

Prioritise backlog→ Build a small slice→ Test & ship→ Measure & gather feedback↻

We already worked this way through the build (one "slice" at a time, each tested and committed). Phase 9 keeps the same rhythm, but now the input is real-world signal — usage, bugs, requests, monitoring alerts — not a pre-written plan.

What is "agile"?

A style of working in small, frequent, feedback-driven increments instead of one big upfront delivery. The point isn't ceremony — it's shortening the gap between "we changed something" and "we learned whether it helped."

What is a "backlog"?

The ordered to-do list for the product — bugs, features, chores, and debt — kept prioritised so the team always knows what's next and why.

3 · Versioning & releases (SemVer)

Once software is out in the world, "which version?" must have a precise answer. The standard is Semantic Versioning — a three-number scheme MAJOR.MINOR.PATCH that communicates the nature of a change.

BumpWhenExample
MAJORA breaking change — existing behaviour/clients would break.1.0.0 → 2.0.0
MINORA new feature, backward-compatible.1.0.0 → 1.1.0
PATCHA bug fix, no new features, backward-compatible.1.0.0 → 1.0.1

Decision for this project: the shipped, smoke-tested MVP is our v1.0.0 — the first release that delivers the promised value. We'll mark it with a git tag, which freezes a named point in history you can always return to or deploy.

# tag the released MVP and push the tag
git tag -a v1.0.0 -m "MVP: booking, reports, referrals, admin — live on Render"
git push origin v1.0.0

What is a "release" vs a "commit" vs a "tag"?

A commit is any saved change. A tag is a permanent label on one specific commit (e.g. v1.0.0). A release is a tagged version you've decided to publish/deploy as a milestone. Tags turn a stream of commits into named, referenceable versions.

What does -SNAPSHOT mean (in pom.xml)?

A Maven convention meaning "in-development, not a fixed release." 0.0.1-SNAPSHOT = a work-in-progress build. A real release drops the suffix (e.g. 1.0.0). It signals the artifact isn't final.

4 · Changelog — v1.0.0

A changelog is a human-readable "what changed, for people" — distinct from the git log, which is "every change, for machines/developers." We follow the Keep a Changelog convention.

Going forward, every release adds a dated section here (Added / Changed / Fixed / Removed / Security) — kept in a CHANGELOG.md at the repo root so it travels with the code.

6 · Tech-debt register

"Tech debt" is a deliberate shortcut taken to ship faster, tracked honestly so it's repaid rather than forgotten. Recording it is what keeps it from becoming a silent liability.

Shortcut takenWhy it was OKThe interest we pay
Booking sends fullName = emailRegistration only captured email; unblocked booking.Patient names are placeholders until a profile step exists.
Report PDFs on local/ephemeral disk (uuid filenames)Simple, worked for MVP; volume persists locally.Lost on Render redeploy; won't scale past one instance.
No pagination on list endpointsTiny data volumes at MVP.Slows and bloats payloads as records grow.
Walk-in phone field collected but not sent to the APIUI built ahead of the field's use.Dead input; mild confusion until wired.
Slot times normalise to UTC on DB read (render "Z")Same instant; frontend localises.Cosmetic inconsistency vs the in-memory "+05:30" form.

Is tech debt bad?

Not inherently — like financial debt, a deliberate, tracked shortcut can be a smart trade to ship sooner. It's only dangerous when it's unacknowledged. The register turns invisible risk into a visible, payable list.

7 · Observability & monitoring

Observability is being able to answer "is the app healthy, and if not, why?" from the outside. Right now, we'd only learn about an outage from a user. That's the first thing to fix in Phase 9.

The three classic signals

SignalQuestion it answersOur starting point
LogsWhat exactly happened, in order?Spring logs to stdout → visible in Render's log stream. Already have an audit_logs table for sensitive actions.
MetricsHow much / how fast / how often? (traffic, latency, errors, memory)None yet. Render shows basic CPU/memory; Spring Boot Actuator can expose more.
TracesWhere did one request spend its time across the system?Not needed at this scale (single service); note for later.

Concrete first steps (cheap, high-value)

  • A health-check endpoint — a lightweight GET /health (or Actuator's /actuator/health) that returns 200 when the app + DB are up. It's the anchor everything else pings.
  • An uptime monitor — a free service (UptimeRobot, Better Stack, or Render's own) pings the health URL every few minutes and emails/texts if it fails. This is how you learn about downtime first. Bonus on a free tier: the ping also keeps the app warm, softening cold starts.
  • Error tracking — a tool like Sentry captures unhandled exceptions with stack traces and context, so you see what broke and how often, not just "something 500'd."
  • Read the logs deliberately — after each deploy, watch Render's log stream for a clean startup (Flyway migrations applied, server listening) before declaring success.

Monitoring vs Observability?

Monitoring = watching known signals for known problems ("is it up? is CPU high?"). Observability = having enough data (logs/metrics/traces) to investigate unknown problems you didn't predict. Monitoring tells you something's wrong; observability helps you find out why.

What is a "health check"?

A trivial endpoint whose only job is to say "I'm alive" (200) or fail. Load balancers, uptime monitors, and orchestrators all rely on it to decide if an instance is healthy.

8 · Maintenance runbook

A runbook is a short, calm checklist for routine operations — so future-you (or a teammate) isn't improvising under pressure.

Shipping a change (the safe loop)

  • Branch → make the change → run tests locally (mvn test, npm test) → open a PR → let CI go green → merge to main.
  • Render auto-deploys main. Watch the deploy log for a clean boot; run a quick smoke check on the live URL.
  • Add a CHANGELOG.md entry; if it's a released milestone, bump the version and tag it.

If the live app is down

  • Check the uptime monitor + Render status/logs → is it a cold start, a crash loop, or the DB?
  • Read the latest deploy log: did a migration fail, or a bad env var slip in?
  • If a recent deploy caused it, roll back to the previous deploy (or the last good tag) first, then diagnose calmly.

Routine hygiene

  • Periodically update dependencies for security patches, letting CI catch regressions.
  • Rotate secrets if ever exposed; never commit them.
  • Keep the backlog honest — add new bugs/requests as they arrive, re-prioritise, and celebrate closing debt.

What is a "rollback"?

Reverting to a previous known-good version when a new deploy misbehaves. It restores service fast, buying you time to diagnose without users suffering. This is exactly why tags and immutable deploys matter.

9 · What I'm learning · next actions

  • 💡 Shipping is the midpoint, not the finish line — the interesting engineering (keeping it healthy, evolving it safely) starts once real users can reach it.
  • 💡 Write the debt down. An acknowledged shortcut is a plan; a forgotten one is a landmine.
  • 💡 You can't fix what you can't see — a health check + uptime monitor is the cheapest, highest-leverage thing to add to a freshly-live app.
  • 💡 Versions are communication — SemVer + a changelog let anyone answer "what's running, and what changed?" without reading code.

Immediate next actions (P0/P1 from the backlog)

  • Tag the MVP as v1.0.0 and add a CHANGELOG.md at the repo root.
  • Harden the production admin credential.
  • Add a health check + uptime monitor so downtime is visible.
  • Decide the first real iteration: persistent report storage, password reset, or the CI-green + monitoring foundation.
