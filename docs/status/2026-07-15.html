The referral-calculation engine

  • Built the configurable rule engine from the Phase-3 design. Rules are scoped by doctor, study, and/or bill amount; when several match, the most specific one wins — scored doctor +4, service +2, amount-range +1.
  • Proved the subtle rule: "most specific wins" beats "highest percentage" — a 15% Thyroid-specific rule correctly out-ranks the 20% default. On completion of a referred visit the payout is written once to a ledger (idempotent), and admins can view per-doctor totals (the top-referrers report) and mark payouts PAID.
  • The scoring logic is a pure function, so it's unit-tested (8 cases) with no database — fast and precise.

Security hardening

🔒 Closed a privilege-escalation hole

Public registration used to accept a role field — meaning anyone could sign themselves up as ADMIN. I found this while reviewing the day's work and fixed it.

  • Public /register now always creates a PATIENT — the role field is gone. STAFF/ADMIN accounts can only be created by an existing admin via POST /api/admin/users, and an initial admin is provisioned at startup.
  • Role-based access everywhere: completing visits & viewing payouts are STAFF/ADMIN; catalogue edits, rule config, user creation & audit logs are ADMIN-only.
  • Refresh tokens + real logout: login now returns a short-lived (15 min) access token plus a long-lived, revocable refresh token (only its hash is stored). Refreshing rotates it; logout revokes it — genuine logout despite otherwise stateless tokens.
  • Report downloads are scoped: a patient can fetch only their own report; others get a 404 (never a 403), so appointment ids can't be probed. Every upload/download and payout is audit-logged.

What I learned

  • "It compiles" proves nothing: the port clash, the timezone rejection, and the privilege hole were all invisible to the compiler — only running the real thing (and reviewing it) surfaced them.
  • Snapshot money at the moment of the transaction: copying each study's price onto the appointment means a later catalogue price change never rewrites an old bill. Same idea as an invoice freezing its line items.
  • Concurrency needs a lock, not just a check: "count bookings, then insert" has a race — two people could grab the last slot. A row lock on the slot serialises them, so capacity holds under load.
  • Cancel is a state change, not a delete: flipping status to CANCELLED keeps the history and frees the slot, where deleting the row would lose the audit trail.
  • Never trust the client's file label: report uploads are checked by both declared type and the %PDF magic bytes, size-capped, and stored with a random name outside any web root.
  • Least privilege by default: the safest role to hand out on self-signup is the weakest one; anything stronger must be granted deliberately by someone who already has it.
  • Push design decisions back to the business: "how close to an appointment can a patient still cancel?" isn't a coding question — I surfaced it and we set a configurable 2-hour rule.

Decisions locked today

  • Cancel/reschedule cutoff: patients may self-change up to 2 hours before the slot (configurable); closer than that they call the clinic.
  • Report storage: PDF only, ≤20 MB, on local disk outside the web root; upload = staff/admin, download = owning patient + staff/admin; every access audited.
  • Access tokens are short-lived (15 min) with 30-day revocable refresh tokens.
  • Consent is captured the moment a patient profile is created (self-booking or walk-in), with a recorded version.
