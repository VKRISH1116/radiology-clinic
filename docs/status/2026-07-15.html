Environment & database

  • PostgreSQL 16 via Docker Compose (postgres:16, named volume clinic_pgdata for durable data, healthcheck on pg_isready). Backend connects over HikariCP; JPA runs ddl-auto=validate, so every entity is checked against the migrated schema at boot (a mismatch fails startup).
  • Flyway migrations are the single source of schema truth, applied on both Postgres and the H2 dev profile:
    V1 12-table schema · V2 seed 9 services + default 20% rule · V3 referring doctors + specificity-demo rules · V4 refresh_tokens.
  • Port-5432 conflict: a native postgresql-x64-17 Windows service was bound to 5432 and shadowing the container on loopback (symptom: FATAL: password authentication failed for user "clinic" — the request was hitting the wrong server). Diagnosed via netstat + a forced-TCP psql version probe, then stopped/disabled the service.
  • JDBC timezone bug: pgjdbc forwards the JVM's TimeZone as a session param; Windows reports IST as the legacy Olson id Asia/Calcutta, which the slim Debian image's tzdata rejects (invalid value for parameter "TimeZone"). Fixed durably by pinning -Duser.timezone=Asia/Kolkata in the Spring Boot Maven plugin.

Auth & security architecture

  • Stateless JWT (HS256, jjwt): claims sub=email, role; 15-min access TTL. A JwtAuthenticationFilter (before UsernamePasswordAuthenticationFilter) parses the Bearer token into the SecurityContext; passwords are BCrypt-hashed.
  • Revocable refresh tokens (V4): opaque 32-byte SecureRandom value, base64url; only its SHA-256 hash is persisted. /refresh validates → rotates (revoke old, issue new); /logout revokes. 30-day TTL. This is how logout works despite stateless access tokens.
  • Method security (@EnableMethodSecurity + @PreAuthorize): roles arrive as ROLE_* authorities; hasRole()/hasAnyRole() gate staff/admin actions.
  • SecurityConfig: SessionCreationPolicy.STATELESS, CSRF disabled (bearer, not cookies), HttpStatusEntryPoint(401) for missing/bad tokens. Earlier fix carried forward: dispatcherTypeMatchers(DispatcherType.ERROR).permitAll() stops thrown 4xx from being re-checked and masked as 403.
  • Owner-scoped 404s: cancel/reschedule and report download return 404 (not 403) when the resource isn't the caller's, so ids can't be probed.

Booking internals

  • Slot grid: 48 × 15-min slots (09:00–21:00 Asia/Kolkata), generated lazily on first request for a date; start_time is UNIQUE (idempotent generation).
  • No double-booking under concurrency: @Lock(PESSIMISTIC_WRITE) — SELECT … FOR UPDATE on the slot row inside the booking tx, then count(status ≠ CANCELLED) < capacity. The lock serialises racers so the check can't be beaten.
  • Price snapshot (BR-9): each appointment_studies.price_snapshot is the service price copied at booking; billed_amount = Σ snapshots (all BigDecimal, never double). Later catalogue edits never rewrite an old bill.
  • Aggregate root: Appointment owns AppointmentStudy via @OneToMany(cascade=ALL); list endpoint uses a left join fetch to avoid an N+1.
  • State transitions, not deletes: BOOKED → CANCELLED / COMPLETED. A configurable ChangeCutoffPolicy (default 2 h) rejects self-changes inside the window with 409.

Referral-calculation engine

Rules filter by scope (doctor / service / amount range; NULL = any). Several can match — a specificity score picks the winner, so "most specific" beats "highest percentage".

// pure, unit-tested — no DB
score = (rule.doctorId  != null ? 4 : 0)
      + (rule.serviceId != null ? 2 : 0)
      + (rule.minAmount != null || rule.maxAmount != null ? 1 : 0);
// highest score wins; tie → first in id order (default rule is id 1)
amount = bill.multiply(pct).divide(100, 2, HALF_UP);   // BigDecimal, 2 dp
  • Verified precedence: a 15% Thyroid rule (score 2) out-ranks the 20% default (score 0) → a ₹1,500 Thyroid visit pays ₹225, not ₹300. A doctor rule (score 4) beats both.
  • Ledger is idempotent (UNIQUE appointment_id) and records the winning rule_id; written on /complete. summariseByDoctor() is a grouped JPQL query for the top-referrers report; payouts flip PENDING → PAID.

Reports, consent & audit

  • Report upload (multipart): validated as a real PDF by content-type and %PDF magic bytes, size-capped 20 MB (Spring multipart + service check), written as {uuid}.pdf under app.reports.dir — outside any web root, random name (no path traversal), git-ignored.
  • Download streams application/pdf with Content-Disposition: attachment; authorised to the owning patient or any staff/admin (else 404).
  • Consent (consent_records) is captured when a patient profile is created (self-booking or walk-in), stamped with app.consent.current-version.
  • Audit (audit_logs): AuditService.record(action, entity, entityId, actor) logs REPORT_UPLOAD, REPORT_DOWNLOAD, PAYOUT_UPDATE. Explicit calls at the sensitive points (an AOP aspect could generalise later).

What I learned

  • "It compiles" proves nothing: the port clash, the timezone rejection and the privilege hole were all invisible to the compiler and to unit tests — running the real stack (and reviewing it) is what surfaced them.
  • Concurrency needs a lock, not a check: "count then insert" races; a pessimistic row lock (SELECT … FOR UPDATE) serialises writers so capacity actually holds.
  • Snapshot money at transaction time (and use BigDecimal + HALF_UP, never double) so historical bills are immutable and exact.
  • Least privilege by default: self-signup should grant the weakest role; elevation must be an explicit action by someone already privileged.
  • Don't trust the client's file label: validate uploads by magic bytes, cap size, randomise the stored name, keep it out of the web root.
  • Push design questions back to the business: the cancel cutoff isn't a coding decision — surfaced it, and we set a configurable 2-hour rule.
