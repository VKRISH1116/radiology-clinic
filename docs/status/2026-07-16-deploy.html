How the live architecture differs from localhost

This was the key learning of the deploy. Locally, one nginx container serves the frontend and reverse-proxies /api to the backend — so the browser only ever talks to one origin and there is no cross-origin problem at all:

browser→ localhost:3000 (nginx)→ /api → backend:8080→ postgres

On Render, the frontend and backend are separate services on separate domains. The browser calls the backend directly from another origin, so two things must be wired up that localhost never needed:

browser→ …frontend-ok1f.onrender.com — cross-origin fetch → …backend-qw3h.onrender.com→ managed PG
  • VITE_API_URL (build-time): the frontend has no proxy now, so it must know the backend's full URL. Vite bakes this into the JS at build — which means changing it requires a rebuild, not just a restart. (This one was already correct — the login request was hitting the right backend URL.)
  • CORS_ALLOWED_ORIGINS (run-time): the backend must explicitly permit the frontend's origin, or the browser blocks the response. This is the one that bit us — see below.

🔒 The one post-deploy bug — CORS preflight 403

Login failed in the browser with a bare Failed to fetch. DevTools showed the real story: the browser's preflight OPTIONS request (which it sends automatically before a cross-origin POST) came back 403 Forbidden with no Access-Control-Allow-* headers — so the browser never sent the actual login POST.

// browser → backend, before the real POST:
OPTIONS /api/auth/login
Origin: https://radiology-frontend-ok1f.onrender.com
Access-Control-Request-Method: POST
   ← backend replied 403 Forbidden (origin not on the allow-list)

// the backend's default was still the localhost dev origin:
allowed-origins: ${CORS_ALLOWED_ORIGINS:http://localhost:5173}

Root cause: Spring Security's CORS filter checks the incoming Origin against a configured allow-list. On Render that list still held only the localhost default, so the real frontend origin was rejected. Fix (no code change — config only): set the backend env var to the exact frontend URL and let Render auto-redeploy:

CORS_ALLOWED_ORIGINS = https://radiology-frontend-ok1f.onrender.com  // exact match, no trailing slash
   → redeploy → preflight 200, login works ✓

This was the expected "chicken-and-egg": each service needs the other's final URL, which only exists after the first deploy — so wiring the pair together always takes one extra redeploy.

⚠️ Free-tier trade-offs (fine for a demo — flagged for honesty)

  • Cold starts: free services spin down after ~15 min idle; the first request then takes ~50 s while the backend wakes. Not a bug — just slow-on-first-hit.
  • Ephemeral report storage: there's no persistent disk, so uploaded report PDFs live only until the next redeploy. Real use needs a paid disk or object storage (S3). The rest of the data is safe — it's in the managed Postgres.

What I learned

  • "Deployed" and "working" are different claims. The build succeeded and the site loaded, yet login was broken — only exercising the real flow through a real browser surfaced it. A smoke test on the live URL is part of deploying, not an afterthought.
  • Same code, different topology, new failure modes. The exact backend that needed no CORS at all locally (one origin behind nginx) suddenly needed it in prod (two origins). The environment, not the code, created the bug.
  • Build-time vs run-time config. VITE_API_URL is frozen into the frontend at build; CORS_ALLOWED_ORIGINS is read at run. Knowing which is which tells you whether a change needs a rebuild or just a restart.
  • Read the preflight, not just the error. "Failed to fetch" says nothing; the OPTIONS 403 in the Network tab named the exact cause. The browser's CORS handshake is where cross-origin problems announce themselves.
  • Blueprint > clicking. Describing the infrastructure in render.yaml makes the deploy repeatable and reviewable — the same discipline as code, applied to servers.
