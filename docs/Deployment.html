1Phases 5–8 recap

P5Build — Frontend

A role-based React 19 + TypeScript SPA (Vite, CSS Modules, React Router), built first against a mock API so screens could be finished before the backend was wired. Three role areas: patient (register/login, dashboard, book, cancel, reschedule), staff (schedule, mark-complete, upload report, walk-in), and admin (catalogue, referral payouts, rules, users, audit). A reusable SlotPicker is shared by the booking and reschedule flows.

P6Integration

Swapped the mock for the live backend over real HTTP, in slices. A small client.ts centralises the fetch layer: base URL, the JWT Bearer header, turning errors into typed ApiErrors, and transparently refreshing the access token once on a 401 then retrying. CORS was enabled on the backend for the browser origin. By the end the mock was deleted — the whole SPA runs on Spring Boot + Postgres.

P7Testing (unit → E2E → UAT)

Three automated layers plus a documented UAT pass. Backend: 79 JUnit/MockMvc tests (REST contracts, role gates, the referral maths, DB rules) on an in-memory H2 profile. Frontend: 20 Vitest + React Testing Library component tests. E2E: one Playwright journey through a real browser. The UAT traceability page maps every PRD acceptance criterion to its evidence — the pass that found (and led us to close) the consent-gate and admin-dashboard gaps.

P8Deployment & CI/CD

The stack is now containerised. Multi-stage Dockerfiles (Maven→JRE for the backend; Vite build→nginx for the frontend) keep the shipped images small. One docker-compose file brings up Postgres + backend + frontend; nginx reverse-proxies /api to the backend, so the browser sees one origin (no CORS, no backend URL baked into the bundle). A GitHub Actions workflow runs the backend and frontend tests + build on every push.

3What's ready

  • Both images build reproducibly; full stack boots with one command.
  • Verified end-to-end in containers: register → login → book → complete → referral, through the proxy.
  • Schema is migration-managed (Flyway V1–V4) and applied automatically on startup.
  • Secrets are externalised: DB password, JWT secret, and the bootstrap-admin password read from a .env file at deploy time (dev defaults only when absent).
  • CI runs backend + frontend tests and the production build on every push.
  • Data survives restarts (named volumes for DB + reports).

4Go-live checklist

Before pointing this at anything real, do these — none is code, all are configuration:

  • Copy .env.example → .env and set strong, unique values for JWT_SECRET (≥32 random bytes), DB_PASSWORD, and BOOTSTRAP_ADMIN_PASSWORD.
  • Set CORS_ALLOWED_ORIGINS to the real frontend URL (e.g. https://clinic.example.com).
  • Put the app behind HTTPS/TLS (a reverse proxy or load balancer terminating certificates) — JWTs and report downloads must not travel over plain HTTP.
  • Change the first admin login immediately after the initial bootstrap.
  • Decide the database home: the bundled db container (single host) or a managed service such as AWS RDS (point DB_URL at it). Set up backups for whichever.
  • Confirm the report-PDF volume is backed up and has enough disk.
Why externalised secrets matter. The dev defaults (a known JWT secret, a known admin password) are fine on a laptop but would be a serious hole in production — anyone reading this public repo could forge tokens. Moving them to an un-committed .env means the code ships without secrets, and each environment supplies its own.

5Commands

Local dev — database only, run backend & frontend from the IDE:

docker compose up -d

Full stack in containers (build + run all three):

docker compose --profile app up -d --build
# open http://localhost:3000

Deploy-style run with real secrets:

cp .env.example .env      # then edit .env with strong values
docker compose --profile app up -d --build

Stop the app (keep the dev DB running):

docker compose --profile app stop backend frontend

6Deploy to Render (chosen PaaS)

The repo has a render.yaml blueprint that declares all three pieces — a managed Postgres, the backend (from its Dockerfile), and the frontend (a static site). On a PaaS the two app pieces are separate services with their own URLs, so the frontend calls the backend cross-origin using VITE_API_URL + CORS (not the local nginx same-origin proxy).

  1. Push to GitHub (done). In the Render dashboard → New → Blueprint, connect this repo. Render reads render.yaml and provisions the database + both services.
  2. It will prompt for the two sync:false values it can't guess: BOOTSTRAP_ADMIN_PASSWORD (choose one you'll remember) — the rest (DB connection, JWT_SECRET) are wired/generated for you.
  3. Let the first deploy finish. Note the two public URLs, e.g. radiology-backend.onrender.com and radiology-frontend.onrender.com.
  4. Wire the two cross-references (chicken-and-egg — each needs the other's URL):
    • Backend env CORS_ALLOWED_ORIGINS = the frontend URL.
    • Frontend env VITE_API_URL = the backend URL.
    Save — Render redeploys the backend and rebuilds the static site with the baked URL.
  5. Open the frontend URL, log in as the bootstrap admin, and change its password.
Verified locally under PaaS-like conditions. The backend was booted with the DB URL assembled from parts, the platform-injected PORT, and no mounted volume — which surfaced (and we fixed) a real bug: the non-root container user couldn't create /data/reports. The image now pre-creates and owns it. Caveat: on Render's free tier there's no persistent disk, so uploaded report PDFs are lost on each redeploy — fine for a demo; attach a disk (paid) or move reports to S3 for real use.
