Full-Stack Web Development — A Field Guide

A ground-up tour of the technologies behind a typical modern web application — for someone who can code but has never shipped a production app. General knowledge, not tied to any one project.

How to read this guide. Every major technology is taught with the same 8-part frame:
  1. What is it? — one sentence.
  2. Why is it used? — the problem it solves.
  3. Analogy — an everyday comparison.
  4. How it's used — where it sits in a real stack.
  5. Remove it? — the consequences of doing without.
  6. Interview answer — a confident 30-second reply.
  7. Beginner mistakes — the classic misunderstandings.
  8. Advanced — what seniors know beyond the basics.

1 · The big picture

A "web app" is software you use through a browser. It always has two halves that talk over the internet: a frontend (runs in the browser, draws the screen) and a backend (runs on a server, holds the real data and rules). Between them sits a database that stores information permanently. Wrapping all of it is infrastructure (where the code runs) and deployment (how it gets there).

Frontend vs Backend vs Database?

Frontend = what you see and click (browser). Backend = the program on a server that enforces the rules and guards the data. Database = the permanent, organised store of that data. The frontend never touches the database directly; it always asks the backend.

What is a "server"?

Just a computer that runs your backend program and answers requests over the network, day and night. In production it's usually a rented cloud machine, not a box in your office.

2 · Three-tier architecture

Almost every serious web app is organised into three tiers, each with one job, each talking only to its neighbour:

   ┌─────────────────────┐
   │  Presentation tier  │   Browser · React frontend · draws UI
   └──────────┬──────────┘
              │  HTTP + JSON (a REST API)
   ┌──────────▼──────────┐
   │  Application tier    │   Backend · business rules · auth
   └──────────┬──────────┘
              │  SQL (via a driver/ORM)
   ┌──────────▼──────────┐
   │  Data tier           │   Database · permanent storage
   └─────────────────────┘

Separating tiers means you can change the UI without touching the database, scale each part independently, and reason about one layer at a time. It's the backbone idea the rest of this guide hangs off.

3 · A request, end to end

Follow one action — a user submitting a form — through every layer. This single story touches most of the technologies in this guide.

  1. Click. The browser holds the form's data in memory (managed by the frontend framework). A click handler fires.
  2. HTTP request. The frontend sends e.g. POST /api/orders with a JSON body and an Authorization header proving who's logged in.
  3. Network. The request travels over HTTPS. If the frontend and backend are on different origins, the browser first sends a CORS preflight to check it's allowed.
  4. Auth. The backend verifies the token/session — who is this, and may they do this? If not: 401/403.
  5. Controller. The backend's web layer receives the request and deserializes the JSON into an object.
  6. Service. Business rules run inside a database transaction.
  7. Repository → DB. The data layer issues SQL; the database writes rows, enforces constraints, commits.
  8. Response. The result is serialized back to JSON with a status code (201 Created) and returned.
  9. Re-render. The frontend updates its state and redraws just the changed part of the screen — no full reload.

Serialize / Deserialize?

Serialize = turn an in-memory object into text (JSON) to send over the network. Deserialize = turn received text back into an object. Frameworks do this automatically at the API boundary.

Transaction?

A group of database changes that must all succeed or all be undone — never half-done. It's what stops you from charging a card but failing to record the order.

Frontend · HTML

HTML

structure

1 · What is it?

The markup language that defines the structure and content of a web page — headings, paragraphs, forms, buttons.

2 · Why used?

The browser needs a description of what is on the page. HTML is that description; everything else (styling, behaviour) attaches to it.

3 · Analogy

The skeleton and organs of a body — the structure everything else is built on.

4 · How it's used

In modern apps you rarely hand-write much HTML; frameworks like React generate it from components. But it's what ultimately reaches the browser.

5 · Remove it?

There is no web page — HTML is the base layer the browser renders.

6 · Interview answer

"HTML is the structural layer of the web — semantic elements that describe content. Frameworks generate it, but it's what the browser ultimately renders and what accessibility and SEO depend on."

7 · Beginner mistakes

Using <div> for everything instead of semantic tags; ignoring accessibility (labels, alt text); confusing HTML (structure) with CSS (looks).

8 · Advanced

Semantic HTML for accessibility/SEO, the accessibility tree, and how the browser parses HTML into the DOM.

Frontend · CSS

CSS

styling

1 · What is it?

The language that controls how HTML looks — colours, spacing, fonts, layout, responsiveness.

2 · Why used?

Structure alone is ugly and unusable on phones. CSS makes it presentable and adapts it to any screen size.

3 · Analogy

Clothing, paint, and interior design applied to the bare structure of a building.

4 · How it's used

Via global stylesheets, component-scoped CSS Modules, utility frameworks like Tailwind, or CSS-in-JS. Layout is done with Flexbox and Grid; theming with custom properties (variables).

5 · Remove it?

Everything renders as unstyled black-on-white text stacked vertically — technically working, practically unusable.

6 · Interview answer

"CSS is the presentation layer. Modern approaches scope it to avoid global collisions — CSS Modules, Tailwind utilities, or CSS-in-JS — and use Flexbox/Grid for layout and custom properties for theming."

7 · Beginner mistakes

Global class-name collisions; fighting the cascade/specificity; fixed pixel layouts that break on mobile; overusing !important.

8 · Advanced

The cascade and specificity model, container queries, prefers-color-scheme for dark mode, and design-system tokens.

Frontend · JavaScript

JavaScript

language · behaviour

1 · What is it?

The programming language browsers run — it makes pages interactive (respond to clicks, fetch data, update the screen).

2 · Why used?

It's the only language browsers execute natively, so every frontend behaviour ultimately runs as JavaScript. It also runs on servers via Node.js.

3 · Analogy

The muscles and nervous system — what makes the structured, styled body actually move and react.

4 · How it's used

Directly, or (more often) authored in TypeScript/React and compiled down to JavaScript. Node.js lets the same language power backends and build tools.

5 · Remove it?

Pages become static documents — no interactivity, no dynamic data, no single-page apps.

6 · Interview answer

"JavaScript is the language of the browser and, via Node, of much tooling and many backends. It's single-threaded with an event loop and async model, which is why non-blocking I/O is central to it."

7 · Beginner mistakes

Confusing JavaScript with Java (unrelated); misunderstanding async/promises; unexpected type coercion (== vs ===); this binding.

8 · Advanced

The event loop, closures, prototypes, microtasks vs macrotasks, and the module system.

What is Node.js?

A runtime that runs JavaScript outside the browser — on servers and command lines. It's what powers npm, build tools like Vite, and JavaScript backends (Express, NestJS).

Frontend · TypeScript

TypeScript

language · JS + types

1 · What is it?

JavaScript with a static type system added — you declare the shape of your data and a checker catches mismatches before the code runs.

2 · Why used?

Large codebases break in subtle ways when data shapes drift. Types catch those bugs at build time and document the code, making refactoring safe.

3 · Analogy

Labelled boxes when moving house — you catch "wrong box" before it smashes, not after.

4 · How it's used

Written like JavaScript with type annotations; a compiler (tsc) checks and erases the types, emitting plain JS the browser runs.

5 · Remove it?

Plain JS still runs, but a whole class of bugs moves from build-time to your users, and large refactors get riskier.

6 · Interview answer

"TypeScript adds compile-time type safety to JavaScript. It catches shape mismatches early, self-documents the domain, and makes refactoring safe. Types are erased at compile time — the browser still runs JavaScript."

7 · Beginner mistakes

Thinking types exist at runtime (they don't); leaning on any; expecting TS to make code faster (it prevents bugs, not slowness).

8 · Advanced

Generics, discriminated unions, narrowing, and encoding invariants so illegal states won't compile.

Compile-time vs run-time?

Compile-time = while you build the app (type checks, bundling). Run-time = while a user is actually using it. Catching a bug at compile-time is cheap; at run-time it's a user-facing failure.

Frontend · React (and its alternatives)

React

UI library

1 · What is it?

A JavaScript library for building UIs from reusable components whose output automatically stays in sync with their data (state).

2 · Why used?

Interactive UIs are hard to keep consistent by hand. React lets you declare "what the screen looks like for this state" and re-renders the right parts when state changes.

3 · Analogy

A spreadsheet: write a formula once, and dependent cells update themselves when inputs change.

4 · How it's used

Components hold state via hooks (useState, useEffect); they compose into pages. Alternatives: Vue, Angular, Svelte. Meta-frameworks like Next.js add routing + server rendering on top of React.

5 · Remove it?

You'd manipulate the DOM by hand — huge, fragile code to keep UI matching data.

6 · Interview answer

"React is a declarative, component-based UI library: UI as a function of state, with efficient re-rendering. It's just the view layer, so routing and data-fetching are separate choices."

7 · Beginner mistakes

Mutating state directly instead of via setters; missing dependency arrays in effects; thinking React is a full framework or that it talks to the DB (it calls an API).

8 · Advanced

The render/reconciliation cycle, memoisation, rules of hooks, and server components / SSR trade-offs.

Component / State / DOM / SPA?

Component = a reusable piece of UI. State = data a component remembers between renders. DOM = the browser's live tree of page elements. SPA (Single-Page App) = a site that loads once and rewrites the screen with JS as you navigate — no full reloads.

Frontend · Tailwind & the styling spectrum

Tailwind CSS

utility-first styling

1 · What is it?

A CSS framework where you style by composing tiny utility classes in your markup (class="flex gap-4 p-6 rounded-lg") instead of writing separate CSS files.

2 · Why used?

It keeps styles co-located with markup, enforces a consistent design scale (spacing, colours), and removes the naming burden of traditional CSS — teams ship UI fast.

3 · Analogy

Lego bricks: small standard pieces you snap together, rather than sculpting each shape from scratch.

4 · How it's used

Add Tailwind to the build; it scans your files and generates only the utilities you used. Alternatives: plain CSS, CSS Modules (scoped files), CSS-in-JS (styled-components), or component libraries (MUI, shadcn/ui).

5 · Remove it?

Nothing breaks — you'd style another way. It's a productivity/consistency choice, not a requirement.

6 · Interview answer

"Tailwind is a utility-first CSS framework — you compose styling from small classes in the markup, with a build step that purges unused utilities. It trades some markup verbosity for speed and design consistency; CSS Modules or CSS-in-JS are common alternatives."

7 · Beginner mistakes

Thinking Tailwind is required for React (it isn't); duplicating long class strings instead of extracting components; forgetting the build must scan your files to include classes.

8 · Advanced

Design tokens via the config, @apply for extraction, and when a component library beats hand-rolled utilities.

Frontend · Vite, npm & the build process

Build tools (Vite / Webpack) · npm

tooling · build

1 · What is it?

A build tool (Vite, Webpack) runs a dev server and bundles your source into optimised static files. npm is the package manager that installs the libraries it all needs.

2 · Why used?

Browsers can't run TypeScript or hundreds of source files efficiently. The build transpiles and bundles everything into a few small files a browser loads fast.

3 · Analogy

A print shop turning a messy manuscript into a compact, finished, readable book.

4 · How it's used

npm install reads package.json and downloads dependencies. npm run dev starts a hot-reloading dev server; npm run build emits a dist/ folder of static assets — that's what you deploy.

5 · Remove it?

No dev server, no TS→JS step, no bundling — you'd ship source the browser can't run.

6 · Interview answer

"Vite is a frontend build tool and dev server; it transpiles TypeScript and bundles the app into optimised static assets. npm manages dependencies from package.json. Users get the built bundle, never the dev server."

7 · Beginner mistakes

Confusing dev server (live tool) with build output (static files); committing node_modules/ or dist/; thinking prod runs npm run dev.

8 · Advanced

Code-splitting, tree-shaking, hashed filenames for cache-busting, and that VITE_-prefixed env vars are inlined at build (not secret).

Transpile / Compile / Bundle / Build artifact?

Transpile = translate one language to another (TS→JS). Compile = turn source into a runnable form. Bundle = combine many files into few. A build artifact is the finished output (a dist/ folder, a .jar) — what you deploy, not your source.

package.json / package-lock.json?

package.json lists dependencies and scripts. package-lock.json pins exact versions so every machine installs identically. npm downloads them into node_modules/.

Backend · Java (and backend language choices)

Java

language · backend

1 · What is it?

A mature, statically-typed, compiled language that runs on the JVM, long favoured for reliable server applications.

2 · Why used?

Its strong typing, huge ecosystem, and first-class transaction/database tooling suit systems where correctness matters (finance, healthcare, enterprise).

3 · Analogy

A diesel truck — not flashy, but dependable for heavy, important loads over years.

4 · How it's used

Usually with the Spring framework, compiled to a runnable .jar. Alternatives: C#/.NET, Node.js/TypeScript, Python (Django/FastAPI), Go, Ruby, PHP — each with its own sweet spot.

5 · Remove it?

You simply pick a different backend language; the architectural ideas here transfer.

6 · Interview answer

"Java on the JVM is a mature, strongly-typed backend choice with excellent transactional and framework support via Spring. It shines where reliability and a large ecosystem matter more than raw novelty."

7 · Beginner mistakes

Confusing Java with JavaScript; confusing the JDK (build) with the JRE (run); thinking it's interpreted (it compiles to bytecode).

8 · Advanced

JVM memory/GC tuning, records and pattern matching, and virtual threads for high-concurrency I/O.

JVM / bytecode / jar / JDK / JRE?

The JVM runs Java. Java compiles to bytecode the JVM executes. A jar is a runnable package of that bytecode. JDK = kit to build Java; JRE = smaller runtime to just run it.

What is a "runtime"?

The environment that executes a program (the JVM for Java; Node or the browser for JS). "At runtime" = while the program is running.

Backend · Spring Boot (Web + Security)

Spring Boot

backend framework

1 · What is it?

A framework that turns Java into a ready-to-run web app with sensible defaults, an embedded server, and modules for web, security, and data.

2 · Why used?

It provides — nearly for free — an HTTP server (Spring Web), JSON handling, database integration, transactions, and a security layer (Spring Security), so you write business logic instead of plumbing.

3 · Analogy

A fully-fitted commercial kitchen versus a bare room — ovens and plumbing already installed.

4 · How it's used

You declare "starters" (web, security, data-jpa, validation) and Spring auto-configures them; it boots an embedded server and manages your objects. Alternatives: Express/NestJS (Node), Django/FastAPI (Python), ASP.NET (C#).

5 · Remove it?

You'd hand-assemble a server, JSON, DB pooling, transactions, and security — weeks of fragile setup.

6 · Interview answer

"Spring Boot gives an embedded server, auto-configuration, dependency injection, and starters for security and persistence. Spring Web handles REST endpoints; Spring Security handles authentication and authorization. It packages to one runnable jar."

7 · Beginner mistakes

Thinking Spring Boot is a language; fighting its conventions; not realising the web server is embedded inside the jar.

8 · Advanced

Auto-configuration internals, profiles for per-environment config, actuator/observability, and reactive (WebFlux) vs servlet stacks.

Backend · Controllers · Services · Repositories

The layered pattern

architecture

1 · What is it?

A convention splitting each backend feature into a Controller (HTTP), a Service (business rules), and a Repository (database access).

2 · Why used?

Separated concerns are testable and maintainable: the controller doesn't know SQL, the service doesn't know HTTP, the repository doesn't know rules.

3 · Analogy

A restaurant: waiter (controller) takes the order, chef (service) applies recipes, pantry keeper (repository) fetches/stores ingredients.

4 · How it's used

Controllers speak in DTOs and status codes; services own transactions and rules; repositories are thin data-access interfaces.

5 · Remove it?

"Fat controllers" mixing HTTP, logic, and SQL — untestable and unmaintainable as they grow.

6 · Interview answer

"I separate controller, service, and repository so HTTP, business logic, and persistence stay decoupled. Rules live in services and are unit-tested independently of the web and database layers."

7 · Beginner mistakes

Business logic in controllers; returning DB entities straight to the client instead of DTOs; transactions in the wrong layer.

8 · Advanced

Transaction boundaries at the service layer; the repository as a swappable port; hexagonal/clean architecture variants.

DTO? Middleware/Filter?

DTO (Data Transfer Object) = an object shaped exactly for the API boundary, keeping public JSON separate from internal entities. Middleware/Filter = code that runs around every request (e.g. authentication, logging) before/after your main handler.

Backend · Dependency Injection

Dependency Injection (DI)

pattern

1 · What is it?

Instead of an object creating what it needs, the framework supplies its dependencies — you declare them and get them handed in.

2 · Why used?

It wires large object graphs automatically and lets you swap real dependencies for fakes in tests — the key to testable code.

3 · Analogy

A hospital assigning you a nurse and equipment rather than making every patient bring their own.

4 · How it's used

Classes are registered (Spring "beans", NestJS providers) and injected via constructors wherever needed. It's an example of Inversion of Control.

5 · Remove it?

You'd construct and pass every object by hand — tangled wiring, and near-impossible to substitute test doubles.

6 · Interview answer

"DI inverts control: the container constructs objects and injects their dependencies, usually via constructors. It decouples classes and makes them trivial to unit-test with mocks."

7 · Beginner mistakes

Treating DI as magic globals; field injection over constructor injection; not knowing beans are singletons by default; circular dependencies.

8 · Advanced

Bean scopes/lifecycles, the IoC container, and composition-root design.

Backend · ORM / JPA / Hibernate

ORM (JPA · Hibernate)

persistence

1 · What is it?

An Object-Relational Mapper maps classes to database tables so you work with objects and it generates the SQL. JPA is Java's standard; Hibernate implements it.

2 · Why used?

It removes mountains of hand-written SQL and result-mapping code, while still allowing custom queries where performance demands.

3 · Analogy

A bilingual translator between two parties who don't share a language — objects on one side, table rows on the other.

4 · How it's used

Entity classes map to tables; repository interfaces provide CRUD + derived queries. Alternatives: query builders (jOOQ, Prisma, Drizzle) or raw SQL. Many teams keep the schema in migrations and let the ORM only validate against it.

5 · Remove it?

Raw database access everywhere: manual SQL, manual mapping, manual connections — far more code and bugs.

6 · Interview answer

"An ORM maps objects to tables and generates SQL, cutting persistence boilerplate. I keep migrations as the source of schema truth and use the ORM in validate mode, dropping to custom queries for hot paths."

7 · Beginner mistakes

The N+1 query problem; letting the ORM auto-create the production schema; storing money as floating-point instead of a decimal type.

8 · Advanced

Lazy vs eager loading, the persistence context/cache, optimistic vs pessimistic locking, and transaction propagation.

Backend · REST & JSON (and API styles)

REST API over HTTP + JSON

API style

1 · What is it?

REST is a convention for web APIs built around resources and HTTP verbs; JSON is the text format the data travels in.

2 · Why used?

The frontend and backend are separate programs needing an agreed contract. REST + JSON is the universal, well-tooled way for them to talk.

3 · Analogy

A standard restaurant menu — everyone knows how to read it and order; you don't invent a new ritual per table.

4 · How it's used

Nouns are resources (/api/orders); verbs are methods — GET read, POST create, PUT/PATCH update, DELETE remove. Status codes carry meaning: 200/201 success, 400 bad input, 401 not logged in, 403 forbidden, 404 not found, 409 conflict, 500 server error. Alternatives: GraphQL, gRPC.

5 · Remove it?

The two halves couldn't communicate without some agreed contract; you'd reinvent an ad-hoc one and lose shared tooling.

6 · Interview answer

"REST models resources as URLs and actions as HTTP verbs, with status codes conveying outcomes. JSON is the payload format. It's stateless and conventional; GraphQL or gRPC fit other needs like flexible querying or high-performance RPC."

7 · Beginner mistakes

Returning 200 for errors; verbs in URLs (/getOrders); ignoring idempotency; leaking internal fields instead of using DTOs.

8 · Advanced

HTTP caching/ETags, pagination, versioning, HATEOAS, and idempotency keys.

What is an API / endpoint / JSON?

An API is the menu of operations a program exposes. An endpoint is one address+verb it answers (POST /api/orders). JSON is human-readable text of key/value pairs and lists, e.g. {"status":"paid","total":2500}.

Backend · Maven vs Gradle

Build tools for the JVM

build · dependencies

1 · What is it?

Tools that download libraries, compile code, run tests, and package a JVM app. Maven and Gradle are the two dominant ones.

2 · Why used?

Real apps need many coordinated libraries at matching versions plus a repeatable build. These tools automate all of it from a config file.

3 · Analogy

A recipe plus a shopping-and-cooking service — the config lists ingredients; the tool fetches them and cooks the dish (the jar).

4 · How it's used

Maven uses XML (pom.xml) and a fixed lifecycle — verbose but predictable. Gradle uses a Groovy/Kotlin script (build.gradle) — more flexible and often faster (incremental builds, caching). Both produce a runnable jar; both resolve dependencies from central repositories.

5 · Remove it?

You'd download dozens of correctly-versioned jars and invoke the compiler by hand — unmanageable.

6 · Interview answer

"Maven and Gradle are JVM build tools. Maven is declarative XML with a rigid lifecycle; Gradle is a programmable build with incremental compilation and better caching. Both manage dependencies and package the app; the choice is largely team preference."

7 · Beginner mistakes

Confusing the two; not understanding dependency scopes (compile/runtime/test); committing build output (target/, build/).

8 · Advanced

Transitive dependency resolution and conflicts, build caching, and multi-module/monorepo builds.

Database · SQL & relational databases

SQL databases (PostgreSQL, MySQL…)

data tier

1 · What is it?

A relational database stores data in tables of rows and columns with enforced relationships; SQL is the language to query and change it.

2 · Why used?

Most business data is relational and needs consistency, transactions, and constraints. Relational DBs guarantee those (ACID) — vital for money and records.

3 · Analogy

A meticulous set of cross-referenced ledgers where every entry must reconcile.

4 · How it's used

Popular engines: PostgreSQL (feature-rich, default choice for many), MySQL/MariaDB, SQLite (embedded), SQL Server. NoSQL alternatives (MongoDB, Redis, DynamoDB) trade strict relations for flexibility/scale in specific cases.

5 · Remove it?

No durable, consistent source of truth; flat files or a mismatched store would corrupt under concurrent use.

6 · Interview answer

"I default to a relational database like PostgreSQL because most domains are relational and need ACID transactions and constraints for integrity. NoSQL fits specific access patterns — caching, documents, huge scale — not as a blanket default."

7 · Beginner mistakes

Storing money as floating-point; skipping foreign keys; no indexes then slow queries; confusing the database (Postgres) with the language (SQL); reaching for NoSQL without a reason.

8 · Advanced

ACID and isolation levels, query planning/EXPLAIN, connection pooling, replication, and sharding.

Tables, rows, columns? SQL vs NoSQL?

A table holds records of one kind (e.g. Users); each row is one record, each column a field. SQL/relational = structured tables with relationships and transactions. NoSQL = looser structures (documents, key-value) for flexibility or scale, usually with weaker guarantees.

Database · Keys, indexes & constraints

Primary/foreign keys · indexes · constraints

data integrity

1 · What is it?

Mechanisms that keep data correct and fast: keys identify and link rows, constraints enforce rules, indexes speed lookups.

2 · Why used?

They push correctness into the database itself, so bad data can't be written even if application code has a bug.

3 · Analogy

A library: the catalogue number (primary key), cross-references between books (foreign keys), and the index at the back (index) that makes finding a topic instant.

4 · How it's used

A primary key uniquely identifies each row; a foreign key forces a column to point at a real row in another table; unique/check constraints enforce rules; an index makes searching a column fast.

5 · Remove it?

Orphaned/duplicate/invalid data creeps in, and queries slow to a crawl as tables grow.

6 · Interview answer

"Primary keys identify rows, foreign keys enforce valid relationships, unique/check constraints enforce rules, and indexes speed reads at a small write cost. I put integrity rules in the database so they hold regardless of app bugs."

7 · Beginner mistakes

No foreign keys "for simplicity"; indexing nothing (slow) or everything (slow writes); assuming an index is free.

8 · Advanced

Composite/covering indexes, B-tree vs hash vs GIN, index selectivity, and reading query plans.

Database · Migrations

Schema migrations (Flyway · Liquibase · Prisma…)

schema versioning

1 · What is it?

Ordered, versioned scripts that build and evolve the database schema, tracked so each runs exactly once.

2 · Why used?

The schema must be identical and reproducible across every environment. Migrations make it code in the repo, not manual clicks someone forgets.

3 · Analogy

Stamped, numbered renovation permits — replay them in order and anyone rebuilds the exact same structure.

4 · How it's used

Tools like Flyway or Liquibase (JVM), or Prisma/Drizzle/Rails migrations, run pending scripts on startup or deploy. The ORM then only validates against the resulting schema.

5 · Remove it?

Schema drift: environments diverge, "works on my machine" bugs appear, and there's no reliable rebuild path.

6 · Interview answer

"Migrations version the schema as ordered scripts in the repo, applied automatically so every environment converges. They're the single source of schema truth; the ORM stays in validate mode."

7 · Beginner mistakes

Editing an already-applied migration (never — add a new one); letting the ORM auto-generate the prod schema; non-portable SQL across engines.

8 · Advanced

Backward-compatible (expand/contract) changes for zero-downtime deploys, checksums, and data migrations vs schema migrations.

Networking · HTTP, HTTPS & DNS

HTTP / HTTPS / DNS / domains

how the web talks

1 · What is it?

HTTP is the request/response protocol browsers and servers speak; HTTPS is HTTP encrypted with TLS; DNS translates a domain name into a server's IP address.

2 · Why used?

They're the plumbing of the web: HTTP carries every API call, HTTPS keeps it private and tamper-proof, DNS lets humans use names instead of numbers.

3 · Analogy

DNS is the phone book (name → number); HTTP is the conversation; HTTPS is speaking in a sealed, private line no one can eavesdrop.

4 · How it's used

A request has a method, URL, headers, and body; a response has a status code, headers, and body. A domain (yourapp.com) resolves via DNS to your host; certificates enable HTTPS.

5 · Remove it?

No HTTP, no web communication. No HTTPS, and traffic (including passwords) is readable by anyone on the network.

6 · Interview answer

"HTTP is the stateless request/response protocol of the web; HTTPS adds TLS encryption; DNS resolves domains to IPs. Every API call is an HTTP request with a method, headers, and body, answered with a status code."

7 · Beginner mistakes

Serving anything real over plain HTTP; confusing a domain (name) with hosting (the server); not understanding HTTP is stateless (hence tokens/sessions).

8 · Advanced

HTTP/2 & HTTP/3, TLS handshakes and certificates, caching headers, and CDNs.

Port / localhost / IP address?

An IP address identifies a machine on a network. A port is a numbered door on that machine so many programs can share one address (web 80/443, dev servers 3000/5173/8080, Postgres 5432). localhost is your own machine addressing itself.

Networking · CORS & allowed origins

CORS (Cross-Origin Resource Sharing)

browser security

1 · What is it?

A browser security rule that blocks a page on one origin from calling an API on a different origin unless that API explicitly allows it.

2 · Why used?

It stops a malicious site from silently using your logged-in session on another site. The API must opt-in by naming which origins may call it.

3 · Analogy

A guest list at a private venue: the door only admits callers the host put on the list.

4 · How it's used

When frontend and backend are on different origins, the backend sets an allowed-origins list. Before a real cross-origin call, the browser sends an automatic preflight OPTIONS request; if the origin isn't allowed, it fails (often 403) and the real call never happens — the classic symptom is a vague "Failed to fetch".

5 · Remove it?

Either the browser blocks all cross-origin calls (broken app), or a careless * weakens a real protection.

6 · Interview answer

"CORS is a browser-enforced policy for cross-origin requests. If the frontend and backend are separate origins, the backend must allow-list the frontend's origin; otherwise the preflight fails and the request is blocked. It's a browser mechanism, not a server-side authorization control."

7 · Beginner mistakes

Thinking it's a backend bug (curl/Postman ignore CORS — only browsers enforce it); trailing-slash/scheme mismatches; using * in production.

8 · Advanced

Preflight caching, credentialed requests, and why CORS is not a substitute for real authorization.

What is an "origin"? A "preflight"?

An origin is scheme + host + port together (https://app.com:443). Two URLs are same-origin only if all three match. A preflight is the browser's automatic OPTIONS "may I?" request before certain cross-origin calls.

Networking · Reverse proxy

Reverse proxy (nginx, etc.)

infra

1 · What is it?

A server that sits in front of your app, receiving all requests and forwarding them to the right backend — and often serving static files itself.

2 · Why used?

It gives one public entry point: it can serve the frontend and route /api to the backend so the browser sees a single origin (no CORS), terminate HTTPS, cache, compress, and load-balance.

3 · Analogy

A building receptionist — visitors talk only to the front desk, which routes them internally.

4 · How it's used

nginx, Caddy, or a cloud load balancer serves the built SPA (with an index.html fallback for client routes) and reverse-proxies API paths to the backend.

5 · Remove it?

You expose the backend directly and serve the SPA separately — reintroducing cross-origin concerns and losing central TLS/caching.

6 · Interview answer

"A reverse proxy is the single front door: it serves static assets, proxies API paths to the backend for a same-origin setup, terminates TLS, and can cache and load-balance. nginx is the common choice."

7 · Beginner mistakes

Forgetting the SPA history fallback (deep-link refresh 404s); confusing a reverse proxy (in front of servers) with a forward proxy (in front of clients).

8 · Advanced

Load-balancing strategies, TLS termination, gzip/brotli, rate limiting, and health checks.

Auth · Authentication, sessions, tokens & JWT

Auth: sessions vs tokens vs JWT

who you are · what you may do

1 · What is it?

Authentication proves who you are (login); authorization decides what you may do. Since HTTP is stateless, apps prove identity on each request with a session or a token.

2 · Why used?

Without it, anyone could read or change anyone's data. Auth gates every protected request.

3 · Analogy

A festival: the ID check at entry (authentication) plus a wristband (session/token) the gates check for each stage you enter (authorization).

4 · How it's used

Session-based: the server stores a session and gives the browser a cookie ID; easy to revoke, needs server state. Token-based (JWT): the server issues a signed, self-contained token the client sends in an Authorization: Bearer header; stateless and cross-origin-friendly. A common pattern: short-lived access JWT + long-lived refresh token (stored hashed so it's revocable). Passwords are always stored hashed (BCrypt/argon2). OAuth/OIDC handles "log in with Google".

5 · Remove it?

An open system: no privacy, no accountability, no way to restrict actions.

6 · Interview answer

"Authentication proves identity; authorization gates actions. I use short-lived JWT access tokens plus revocable, hashed refresh tokens — stateless and cross-origin friendly. Passwords are hashed with BCrypt. Sessions are the stateful alternative when server-side revocation is a priority."

7 · Beginner mistakes

Confusing authentication with authorization; thinking a JWT is encrypted (it's only signed — readable, so no secrets inside); storing plaintext passwords; long-lived tokens with no revocation.

8 · Advanced

Token rotation/replay detection, HS256 vs RS256, the stateless-vs-revocable trade-off, secure cookie flags, and OAuth2/OIDC flows.

Hashing? Cookie? Bearer token?

Hashing = a one-way scramble so the DB never stores the real password. A cookie is a small value the browser stores and sends back automatically (used for sessions). A Bearer token is a credential sent in the Authorization header — "whoever bears it, gets access", so protect it.

Infra · Docker (images, containers, volumes)

Docker

containerisation

1 · What is it?

A tool that packages an app with everything it needs into a portable image, which runs as an isolated container identically anywhere.

2 · Why used?

It kills "works on my machine": the same image runs on your laptop, a teammate's, CI, and production — no fiddly per-machine setup.

3 · Analogy

A shipping container — pack once, and any ship, port, or truck handles it identically regardless of contents.

4 · How it's used

A Dockerfile (recipe) builds an image; running it gives a container. Multi-stage builds compile in a fat toolchain image, then ship only the artifact on a slim runtime image. Volumes persist data outside containers; networks let containers talk; ports are published to the host.

5 · Remove it?

Every machine needs the exact tools installed and matched by hand; deploys become bespoke and fragile.

6 · Interview answer

"Docker containerises an app with its dependencies into an image that runs identically everywhere. I use multi-stage builds to keep final images small and run as a non-root user. Volumes persist stateful data outside the container."

7 · Beginner mistakes

Confusing an image (template) with a container (running instance); shipping the build toolchain in the final image; running as root; assuming container data persists without a volume.

8 · Advanced

Layer caching, image size/attack surface, .dockerignore, and orchestration (Kubernetes).

Image vs container vs Dockerfile vs volume?

Dockerfile = recipe. Image = frozen template built from it. Container = a live, isolated instance of an image (one image → many containers). Volume = storage that lives outside the container so data survives restarts/rebuilds.

Infra · Docker Compose

Docker Compose

multi-container orchestration (dev)

1 · What is it?

A tool to define and run a set of containers together from one docker-compose.yml.

2 · Why used?

A typical app is several cooperating containers (db + backend + frontend). Compose starts them on a shared network, in order, with one command.

3 · Analogy

A stage manager cueing the whole cast to enter in the right order.

4 · How it's used

Each service names its image/build, ports, env, and volumes; containers reach each other by service name on a private network. Healthchecks and depends_on order startup.

5 · Remove it?

You'd start, network, and order each container by hand every time.

6 · Interview answer

"Compose orchestrates a local multi-container stack from one file — networking, dependency ordering, healthchecks, and volumes. It's ideal for dev and single-host; production scale uses Kubernetes or a PaaS."

7 · Beginner mistakes

Using localhost between containers (they address by service name); not waiting for the DB to be healthy; wiping volumes with down -v unintentionally.

8 · Advanced

Profiles for optional services, override files per environment, and the step up to Kubernetes for multi-host scaling.

Deploy · Where apps run: Render · Railway · Vercel · Netlify · the cloud

Hosting & PaaS platforms

deployment

1 · What is it?

Platforms that run your app on rented cloud machines so you don't operate servers yourself — you push code or a config file and they host it.

2 · Why used?

Provisioning servers, networking, TLS, and databases by hand is a lot. A PaaS handles it, so a small team can ship to the internet fast, often free to start.

3 · Analogy

Renting a serviced apartment (PaaS) vs building a house (raw cloud servers) — you move in and live; the landlord maintains the building.

4 · How it's used

Pick by workload — see the table. Full-stack PaaS (Render/Railway/Fly) can run backends + managed databases; frontend-focused hosts (Vercel/Netlify) excel at static sites and serverless functions; raw cloud (AWS/GCP/Azure) gives maximum control via IaaS.

5 · Remove it?

You'd provision and operate servers, TLS, and databases yourself — far more to learn and maintain for an MVP.

6 · Interview answer

"I deploy on a PaaS described declaratively (e.g. Render's blueprint or a Railway/Vercel config): managed database, a containerised or buildpack backend, and a static frontend. The platform injects secrets and handles TLS. Raw AWS is the option when you need full control."

7 · Beginner mistakes

Expecting local same-origin behaviour in prod (CORS surfaces); ignoring free-tier limits (cold starts, no persistent disk); setting build-time vars but forgetting to rebuild.

8 · Advanced

The PaaS↔IaaS spectrum, blue-green/rolling deploys, secrets management, autoscaling, and moving stateful bits to managed services (S3, RDS).

PlatformBest at · notes
VercelFrontend/Next.js + serverless functions; superb DX for React; not for long-running backends.
NetlifyStatic sites + functions; similar niche to Vercel.
RenderFull-stack PaaS: web services, static sites, managed Postgres, cron; blueprint file.
RailwayFull-stack PaaS with a slick UX; databases + services; usage-based.
Fly.ioRun containers close to users globally; good for backends.
AWS / GCP / AzureRaw cloud (IaaS/PaaS mix): maximum power and complexity; the long-term "grow into it" option.

Deploy · Environments & environment variables

Dev vs Prod · env vars · secrets

configuration

1 · What is it?

Environments are separate copies of your app (development, staging, production). Environment variables are config values supplied from outside the code so the same build behaves differently in each.

2 · Why used?

URLs, database credentials, and secrets differ per environment and must never be hard-coded or committed. Env vars keep code identical and config external.

3 · Analogy

The same appliance with a voltage switch — one device, adapted to the country it's plugged into.

4 · How it's used

Set in a git-ignored .env locally and in the platform's dashboard in prod. Key distinction: build-time vars are frozen into the artifact (e.g. a frontend API URL — changing needs a rebuild); run-time vars are read at startup (e.g. DB password, signing key — a restart suffices).

5 · Remove it?

You'd hard-code secrets and URLs, leaking credentials into git and breaking every time you move environments.

6 · Interview answer

"Config lives in environment variables, not code — different values per environment, secrets injected by the platform and kept out of git. I'm careful about build-time vs run-time vars: the former are baked into the build and need a rebuild to change."

7 · Beginner mistakes

Committing .env with secrets; assuming a changed build-time var takes effect without rebuilding; mixing up dev and prod credentials.

8 · Advanced

Secret managers (Vault, AWS Secrets Manager), config precedence, and the twelve-factor config principle.

Dev · Git & GitHub

Git · GitHub

version control

1 · What is it?

Git records your code's full history as commits; GitHub hosts that repository online for collaboration.

2 · Why used?

You get a safe, revertible history, parallel work via branches, and a review workflow via pull requests — the foundation of team development.

3 · Analogy

A document's full track-changes history with named save points you can jump back to.

4 · How it's used

You commit snapshots on a branch, push them to GitHub, and open a pull request to merge — where teammates review and automated checks run before it lands on main.

5 · Remove it?

No history, no safe rollback, no collaboration — a single fragile copy of the code.

6 · Interview answer

"Git is distributed version control; GitHub hosts it and adds pull requests and CI. I work on branches, commit small logical changes, and merge via reviewed PRs with required checks so main stays releasable."

7 · Beginner mistakes

Committing straight to main for risky work; committing secrets or build output; confusing a commit (local save) with a push (upload); giant unfocused commits.

8 · Advanced

Rebase vs merge, branch protection, trunk-based vs GitFlow, and bisecting to find regressions.

Commit / branch / push / pull request?

Commit = a saved snapshot with a message. Branch = an independent line of work. Push = upload commits to GitHub. Pull request (PR) = a proposal to merge a branch, where it's reviewed and CI-checked first.

Dev · CI/CD

CI/CD (GitHub Actions, etc.)

automation

1 · What is it?

Continuous Integration automatically builds and tests every change; Continuous Delivery/Deployment automatically ships it onward once it's green.

2 · Why used?

It catches broken changes in minutes instead of in production and makes releases repeatable instead of manual and error-prone.

3 · Analogy

An automatic proofreader-and-publisher that checks every draft the moment you submit it, then prints it if it passes.

4 · How it's used

A pipeline file (e.g. .github/workflows/ci.yml) defines jobs that run on every push/PR: install, build, run tests, and optionally deploy. Tools: GitHub Actions, GitLab CI, CircleCI, Jenkins.

5 · Remove it?

No automated safety net: regressions slip through and deploys become manual and inconsistent.

6 · Interview answer

"CI runs the build and test suites on every push so regressions are caught before merge; CD automates the release once checks pass. I keep main always releasable behind required status checks."

7 · Beginner mistakes

Thinking CI auto-deploys by default (it tests; deploy is a separate step); flaky tests eroding trust; no caching so pipelines crawl.

8 · Advanced

Build caching, matrix builds, environment-scoped deploys, required checks + branch protection, and progressive delivery.

Reference · Mock 10-minute technical presentation

[0:00 – Shape] "A modern web app is three tiers: a browser frontend, a server backend, and a database — connected by an HTTP+JSON API. Each tier has one job and talks only to its neighbour."

[1:30 – Frontend] "The frontend is a single-page app, typically React + TypeScript, built by a tool like Vite into static assets. It holds UI state, renders declaratively, and calls the backend over HTTP — it never touches the database."

[3:00 – Backend] "The backend — say Spring Boot on Java, or Express on Node — exposes a REST API. Internally it's layered: controllers handle HTTP, services hold business rules and transactions, repositories talk to the database, usually through an ORM. Dependency injection wires it together and makes it testable."

[4:30 – Data] "State lives in a relational database like PostgreSQL, chosen for ACID transactions and integrity via keys and constraints. The schema is versioned as migrations in the repo, so every environment is reproducible."

[6:00 – Security & networking] "Auth proves identity per request — commonly short-lived JWTs plus refresh tokens, with hashed passwords. Everything runs over HTTPS. When frontend and backend are separate origins, CORS must allow-list the frontend; a reverse proxy can instead present a single origin."

[7:30 – Delivery] "The app is containerised with Docker so it runs identically everywhere; Compose orchestrates it locally. In production a PaaS like Render, Railway, or Vercel hosts it, with secrets injected as environment variables. CI/CD runs tests on every push and ships green builds."

[9:00 – Judgement] "The interesting decisions are trade-offs: SQL vs NoSQL, sessions vs tokens, single-origin proxy vs CORS, PaaS vs raw cloud. Knowing why you chose each — not just that it works — is what separates shipping from understanding."

Reference · The "CEO explanation" (zero jargon)

"Think of a web app as a shop with three parts.

The shopfront is what customers see and touch in their browser — the buttons, forms, and pages. It's designed to be pleasant and responsive, but it doesn't actually hold anything valuable.

Behind it is the back office — a program running on rented computers in the cloud. It enforces all the rules ("only this customer can see their own order"), does the real work, and is the only part allowed to open the safe.

The safe is the database: a secure, highly-organised store of all the important information — customers, orders, records — kept consistent so nothing is ever lost or contradictory.

Between the shopfront and back office runs a constant, private, encrypted conversation. Guards on the door check ID so only the right people get in, and only to the areas they're allowed.

We build all this in small, reversible steps that are automatically checked for mistakes before going live, and it runs on rented cloud computers so there's no hardware to maintain and it can grow as demand grows.

In one line: a friendly shopfront, a rule-enforcing back office, and a secure safe — connected privately, guarded carefully, and shipped in safe, automated steps."
