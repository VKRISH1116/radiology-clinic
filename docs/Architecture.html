1System overview — a 3-tier web app

The app is a classic three-tier architecture: a browser UI, a server that holds the logic, and a database. Each tier has one job and talks to the next through a clear contract.

1 · Presentation

React (browser)

Screens, forms, routing. Talks to the server over HTTPS with JSON. Holds no business rules.

→

2 · Application

Java · Spring Boot (REST API)

Authentication, business rules (billing, referral engine), validation. The only tier that touches the DB.

→

3 · Data

PostgreSQL

Durable storage: patients, appointments, reports metadata, referral rules & ledger.

Why separate tiers? So each can change independently and be secured separately. The browser is untrusted (anyone can tamper with it), so all rules and access checks live on the server — the React app is a convenient face, never the source of truth. Report files live on disk/object storage; the database stores only their metadata and who may access them.

2Architecture Decision Records — the "why"

Each ADR: the decision, why it fits this project, and the alternatives we consciously rejected.

ADR-1Database: PostgreSQL accepted

Why Our data is highly relational (patients→appointments→studies→referrals) and needs transactions and accurate money math (NUMERIC, not floats). Postgres is free, open-source, rock-solid, and the default choice for Spring Boot.
Rejected Oracle — powerful but commercial/costly, overkill for one clinic. MySQL — fine, but Postgres has richer types (JSON, enums, checks) and stricter SQL. MongoDB — document DBs are weak for the many-to-many, transactional relations at the heart of referrals.
Trade-off You must design a schema up front (vs Mongo's schema-less speed) — but that rigor is exactly what protects patient and payout data.

ADR-2Backend: Java + Spring Boot accepted

Why A constraint (C1), and a great fit: Spring Boot gives batteries-included security, data access (JPA), validation, and REST with minimal boilerplate. Huge ecosystem, very employable skill.
Rejected Plain Java servlets — you'd hand-build everything Spring gives free. Node/Express — lighter, but Java/Spring is the stated learning target and stronger for typed, long-lived business systems.
Trade-off More initial concepts to learn (beans, annotations) — but they encode industry best-practice you want to know.

ADR-3Frontend: React accepted

Why A constraint (C1). React's component model and state management make the multi-screen, role-based UI maintainable, with a massive ecosystem (routing, forms).
Rejected Vanilla JS — fine for a page or two, but manual DOM updates become unmanageable across many stateful screens. Angular — heavier/opinionated; steeper for a first project.
Trade-off A build toolchain (Vite/npm) to learn — worth it for componentised, testable UI.

ADR-4API style: REST over JSON accepted

Why Simple, universal, easy to learn and debug (works with a browser + curl). Maps cleanly to our resources (appointments, reports, doctors).
Rejected GraphQL — flexible querying we don't need yet; adds complexity. gRPC — great for service-to-service, awkward from browsers.

ADR-5Auth: JWT (stateless tokens) accepted

Why A React SPA + REST API pairs naturally with JWTs: the server issues a signed token at login; the browser sends it on each request; the server verifies it without server-side session storage. Carries the user's role for authorization.
Rejected Server sessions + cookies — perfectly valid, but stateful and slightly more setup for an SPA. We can revisit if we need instant token revocation.
Trade-off Tokens can't be individually revoked before expiry — mitigated with short lifetimes + refresh tokens later.

ADR-6Backend shape: layered (Controller → Service → Repository) accepted

Why Separation of concerns: Controllers handle HTTP, Services hold business rules, Repositories talk to the DB. Testable, and the standard Spring pattern.
Rejected Putting SQL/logic directly in controllers — fast to write, painful to test and maintain.

3Backend layers & request flow

A single request — "download my report" — flows through the layers like this:

Browser (React)
  │  GET /api/appointments/42/report   (Authorization: Bearer <JWT>)
  ▼
[Security filter]  verify JWT, load user + role
  ▼
[Controller]       ReportController.download(42)         ← HTTP in/out only
  ▼
[Service]          ReportService: is this MY report? (access rule NFR-01) → audit log
  ▼
[Repository]       ReportRepository.findByAppointmentId(42)   ← Spring Data JPA
  ▼
PostgreSQL         SELECT ... FROM reports WHERE appointment_id = 42
  ▲
  └─ file streamed back if authorised, else 403 (logged)
Learning note. Notice the access check lives in the Service, not the UI. Even if someone crafts the request by hand (no browser), the server still enforces "only the owner or staff can read this report." Security you can only bypass by compromising the server is real security.

5Referral engine (the heart of the app)

When an appointment is marked Completed and has a referring doctor, the engine computes the payout and writes one referrals row. Rules are matched, then the most specific wins (BR-3).

function computeReferral(appointment):
    if appointment.referring_doctor is null: return   // no referral
    bill = appointment.billed_amount
    candidates = referral_rules where active AND
        (doctor_id is null OR = appointment.doctor) AND
        (service_id is null OR matches a study in the visit) AND
        (min/max is null OR bill within range)
    // specificity score: doctor(+4), service(+2), amount-range(+1)
    winner = candidates.maxBy(specificityScore)   // default 20% rule scores 0
    amount = round(bill * winner.percentage / 100, 2)
    upsert referrals(appointment, winner.doctor, winner.id, amount, PENDING)
Worked example (from PRD AC-F6-2). Bill ₹2,000, Dr. Rao. Rules: default 20% (score 0) and "Dr. Rao = 25%" (score 4). Winner = the doctor rule → ₹2,000 × 25% = ₹500. With only the default rule present, it would be ₹400. This matches the acceptance criteria exactly — design tracing straight back to the PRD.

7Security & privacy design

  • Passwords stored as BCrypt hashes, never plaintext (one-way, salted).
  • JWT signed by the server; carries user id + role; short expiry. Sent as Authorization: Bearer.
  • Authorization enforced server-side per endpoint by role, plus ownership checks (a patient can only read their data — NFR-01).
  • Consent gate: registration writes a consent_records row; no personal data persists without it (NFR-02).
  • Audit log for report access & payout changes (NFR-03).
  • Transport: HTTPS in deployment; secrets (DB password, JWT key) in env vars, never in git (already in .gitignore).
  • Report files stored outside the web root; served only through the access-checked endpoint.

Why organise by feature (patient/, appointment/, referral/) not by layer? Grouping a feature's controller+service+repo together keeps related code close — easier to navigate and grow than one giant controllers/ folder. This is "package by feature", a widely-recommended Spring practice.
